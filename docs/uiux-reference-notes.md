# UI/UX Reference Notes

무슨일의 UI는 공개 정보 대시보드, 지도형 탐색, 출처 비교 화면의 공통 패턴을 참고하되, 댓글/투표/동원/자극적 속보 UX는 제외한다.

## 참고한 방향

- 공개 정보 대시보드: 출처와 근거를 분리해 반복 확인 가능한 구조.
- 지도형 탐색 UI: 위치, 범위, 경로를 한 화면에서 비교하는 구조.
- 이슈 트래커 상세 UI: 요약, 흐름, 근거, 다른 주장을 탭으로 분리하는 구조.

## 무슨일 적용

- 첫 화면은 마케팅 랜딩이 아니라 홈/지도/상세/제보가 동시에 보이는 운영 화면이다.
- 첫 판단 단위는 개별 상황 카드가 아니라 이슈다. 사용자는 이슈를 먼저 고르고, 그 안에서 지역·시간·교통·인파·경로 상황을 내려본다.
- 지도는 중앙 주 화면이고, 좌측은 이슈와 관련 상황, 우측은 객관 요약과 제보다.
- 각 상황은 단일 진실 Event가 아니라 target card + 출처별 자료로 보인다.
- 같은 이슈의 하위 대상은 Occurrence, ContinuousPresence, TransitOccurrence, CrowdDensitySignal, RouteSegment, RouteCheckpoint로 분리해 표시한다.
- 기본 목록은 현재/예정 자료를 기록/통계 자료보다 먼저 보여준다.
- 과거 일정과 통계는 `기록` 필터에서 명시적으로 다시 볼 수 있어야 한다.
- 확인 가능한 지역 현황은 보조 정보이며, 실제 상황 목록보다 먼저 시선을 차지하지 않는다.
- 상태 칩은 `진행 중`, `시간 확인 중`, `쟁점`, `권리 검토`처럼 불확실성을 보존한다.
- 제보는 텍스트 제목 입력이 아니라 유형과 Proof-of-Presence 조건 중심이다.
- 알림 CTA는 “모든 업데이트”가 아니라 기본적으로 의미 있는 변화만 따른다.
- 후원, 자유 댓글, 추천/비추천, 찬반투표는 UI에 없다.

## 품질 기준

- 모바일에서 한 화면에 홈/지도/제보 탭이 남아 있어야 한다.
- 모바일 하단 탭은 실제 이동 가능한 화면만 제공한다.
- 카드 안 텍스트가 넘치지 않아야 한다.
- 색만으로 상태를 구분하지 않고 텍스트 칩을 함께 쓴다.
- 색상은 `primary`, `official`, `pending`, `dispute`, `risk`, `transit`, `crowd`, `route`, `archive` 의미 토큰으로만 상태를 표현한다.
- 같은 의미는 홈 카드, 상세, 지도, 배지에서 같은 색을 쓴다.
- 지도 패널은 API와 지도 타일이 없어도 비어 보이지 않아야 한다.
- 상세 화면은 요약, 흐름, 근거, 다른 주장을 분리한다.
- 사용자 원문은 UI에 직접 표시하지 않는다.

## 객관성 QA

홈, 카드, 지도 시트, 상세 화면은 다음 기준을 통과해야 한다.

- 확인된 내용이 보인다.
- 아직 확인 중인 부분이 보인다.
- 다른 주장이 있거나 없다는 상태가 보인다.
- 확인 정도가 보인다.
- 사용자 원문, 정밀 위치, 얼굴, 차량번호, 주소, 전화번호가 보이지 않는다.
